## ✅ 라우팅 경로 누락 가능성 상세 설명

### 🧭 문제 개요

AKS에서 Oracle DB(예: On-Prem, ExpressRoute 연결)에 연결하려면 AKS의 Pod → VNet → ExpressRoute → On-Prem 네트워크 → DB 로의 \*\*올바른 경로(Route Path)\*\*가 설정되어 있어야 합니다.

`traceroute` 명령에서 `* * *` 구간이 중간 또는 마지막에서 반복될 경우 다음 원인을 의심할 수 있습니다:

| Hop 위치            | 원인 가능성                                |
| ----------------- | ------------------------------------- |
| **초반 (1\~3 hop)** | AKS VNet/NSG/라우팅 문제                   |
| **중간 (\~6 hop)**  | ExpressRoute 경유 실패, Peering 또는 UDR 누락 |
| **마지막 hop 없음**    | On-Prem 방화벽/ACL 또는 DB 서버 문제           |

## 🔎 점검 항목 및 스크립트

### 1️⃣ **UDR 확인**: AKS 서브넷에 지정된 라우팅 테이블이 ExpressRoute로 향하는지 확인

#### 🔧 설명

* AKS Subnet은 기본적으로 Azure System route를 따르지만, 사용자 정의 라우트(User Defined Route, UDR)가 지정되어 있을 경우 이를 따라갑니다.
* 이 라우트가 ExpressRoute의 게이트웨이 IP로 향하는지 확인해야 합니다.

#### ✅ 확인 스크립트 (Azure CLI)

```bash
# 변수 설정
AKS_RG="your-aks-resource-group"
AKS_VNET="your-aks-vnet-name"
AKS_SUBNET="your-aks-subnet-name"

# Subnet 정보 확인
az network vnet subnet show \
  --resource-group $AKS_RG \
  --vnet-name $AKS_VNET \
  --name $AKS_SUBNET \
  --query "{SubnetName:name, RouteTable:id}" \
  --output table
```

→ 결과에서 `RouteTable` 값이 있다면, 해당 UDR을 추출하여 다음을 확인:

```bash
# UDR 라우팅 테이블의 실제 경로 확인
UDR_NAME="your-udr-name"
UDR_RG="your-udr-rg"

az network route-table route list \
  --resource-group $UDR_RG \
  --route-table-name $UDR_NAME \
  --output table
```

👉 `NextHopType`이 `VirtualNetworkGateway` 또는 `VirtualAppliance` 여야 하고, IP가 ExpressRoute Gateway 또는 NVA일 것

---

### 2️⃣ **VNet Peering 라우팅 허용 여부 확인**

#### 🔧 설명

* AKS가 있는 VNet과 ExpressRoute Gateway가 연결된 VNet 간 Peering이 존재해야 하며,
* Peering 설정에서 `Use Remote Gateway`, `Allow Gateway Transit` 등이 설정되어야 합니다.

#### ✅ 확인 스크립트

```bash
az network vnet peering list \
  --resource-group $AKS_RG \
  --vnet-name $AKS_VNET \
  --query "[].{Name:name, AllowForwardedTraffic:allowForwardedTraffic, UseRemoteGateways:useRemoteGateways, AllowGatewayTransit:allowGatewayTransit}" \
  --output table
```

→ 결과 예시:

| Name       | AllowForwardedTraffic | UseRemoteGateways | AllowGatewayTransit |
| ---------- | --------------------- | ----------------- | ------------------- |
| to-gw-vnet | True                  | True              | False               |

* `UseRemoteGateways=True` 가 **AKS 쪽 Peering 설정**에 있어야 함
* `AllowGatewayTransit=True` 는 **ExpressRoute Gateway가 연결된 VNet 쪽 Peering 설정**에 있어야 함

---

### 3️⃣ **ExpressRoute Gateway 라우팅 확인**

#### 🔧 설명

* ExpressRoute 연결이 올바르게 설정되었는지, On-Prem 경로가 Gateway에서 받아지는지 확인해야 합니다.

#### ✅ 라우팅 테이블 확인 (NVA or Gateway에서 필요, CLI로는 한계 있음)

Azure에서 직접 `effective routes` 조회:

```bash
az network nic list --query "[].{Name:name,ID:id}" -o table
```

→ AKS 노드 VM의 NIC를 찾아서:

```bash
az network nic show-effective-route-table \
  --resource-group NODE_RESOURCE_GROUP \
  --name NODE_NIC_NAME \
  --output table
```

→ On-Prem IP 대역(예: `10.241.0.0/16`) 에 대해 **NextHopType = VirtualNetworkGateway** 이 나와야 함

---

### 4️⃣ **traceroute 해석 기준**

```bash
traceroute -n 10.241.185.68
  172.31.32.133
```

* 예시 (정상):

  ```
  1  172.23.0.1     # AKS VNet
  2  10.10.10.1     # ExpressRoute Gateway
  3  10.241.1.1     # On-Prem Router
  4  10.241.185.68  # Oracle DB
  ```

* 예시 (비정상):

  ```
  1  172.23.0.1
  2  *
  3  *
  4  *
  ```

→ `*` 발생 위치 기준으로 문제 위치 추정

## 📌 추가 팁

* 라우팅 점검 외에도 다음 항목 병행 권장:

  * AKS Node Outbound IP 확인 → 방화벽 IP 허용 리스트에 등록 여부 확인
  * AKS Subnet NSG 룰에 outbound 규칙 허용 여부 확인 (`TCP 1521`, `Destination: 10.241.185.68`)
  * DB 서버 ACL 설정 확인 (특정 IP만 허용하는지)
